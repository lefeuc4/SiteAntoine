## Architecture Patterns

### Next.js App Router Special Files

**Pattern: error.tsx (error boundary)**
- Must be a Client Component (`'use client'`)
- Receives `error: Error & { digest?: string }` and `reset: () => void` props
- Placed at route-group level: `src/app/(frontend)/error.tsx` catches all errors in all frontend routes
- When a Server Component throws (e.g., `getPayload()` connection failure), Next.js renders the nearest `error.tsx`
- The `reset()` function attempts to re-render the failed segment — wire it to a "Reessayer" button

```typescript
// src/app/(frontend)/error.tsx
'use client'

export default function Error({
  reset,
}: {
  error: Error & { digest?: string }
  reset: () => void
}) {
  return (
    <main className="max-w-[1280px] mx-auto px-8 lg:px-12 py-24 text-center">
      <p className="text-base font-body text-gris-ardoise mb-6">
        Contenu temporairement indisponible. Actualisez la page ou revenez dans quelques instants.
      </p>
      <button
        onClick={reset}
        className="bg-bleu-electrique text-blanc-pur rounded-full px-6 py-3 font-heading text-sm hover:brightness-110 transition-all"
      >
        Reessayer
      </button>
    </main>
  )
}
```

**Pattern: loading.tsx (Suspense boundary)**
- Server Component (no `'use client'` needed)
- Placed at route-group level: `src/app/(frontend)/loading.tsx`
- Shown while page Server Components are executing (Payload query in progress)
- For Neon PostgreSQL cold starts (2-5s), this prevents blank-screen hang
- Keep minimal — a centered spinner or pulsing placeholder matching brand

```typescript
// src/app/(frontend)/loading.tsx
export default function Loading() {
  return (
    <main className="max-w-[1280px] mx-auto px-8 lg:px-12 py-24 flex justify-center">
      <div className="animate-pulse text-gris-ardoise text-base font-body">
        Chargement...
      </div>
    </main>
  )
}
```

**Pattern: not-found.tsx (404 page)**
- Server Component
- Placed at route-group level: `src/app/(frontend)/not-found.tsx`
- Renders branded 404 experience when `notFound()` is called or route doesn't match
- Include nav back to Accueil

Source: Next.js App Router — Error Handling docs (https://nextjs.org/docs/app/building-your-application/routing/error-handling)

### Design Token Fix Pattern

The header `bg-white` → `bg-blanc-pur` fix is a one-line change in `Header.tsx`. The `bg-white` Tailwind utility maps to `#FFFFFF`; the spec token `bg-blanc-pur` maps to the `--color-blanc-pur: #F8FAFC` value in `globals.css`. Tailwind v4 custom `@theme` tokens are available as utilities automatically.

```typescript
// Header.tsx line 29 — before
const bgClass = scrolled ? 'bg-white shadow-sm' : 'bg-white'

// Header.tsx line 29 — after
const bgClass = scrolled ? 'bg-blanc-pur shadow-sm' : 'bg-blanc-pur'
```

### Timeline Year Label Pattern

The seeded `titre` values already contain the year as a prefix: `"2010 — Les Premiers Pas"`. The spec requires a separate year label rendered as `text-sm font-body text-gris-ardoise`. The correct approach is to parse the title in the rendering component, not to change the Payload schema.

**Why no schema change:** The `PageContent.ts` collection has `create: () => false` (D-12 decision — structure fixed, seed only). Adding a new field would require a migration and reseed. Parsing the existing string is lower risk and aligns with the "no schema changes" constraint.

**Parsing strategy:** Split on ` — ` (space-em-dash-space). If the first segment looks like a 4-digit year, render it as the year label; render the rest as the title. Fallback: if no match, render the full `titre` as title with no year label.

```typescript
// mon-histoire/page.tsx — parse titre for year label
function parseTitre(titre: string): { annee: string | null; nom: string } {
  const match = titre.match(/^(\d{4})\s*[—–-]\s*(.+)$/)
  if (match) return { annee: match[1], nom: match[2].trim() }
  return { annee: null, nom: titre }
}
```

### Typography Token Corrections

The contact page uses `text-[1.75rem]` and `text-[2.25rem]` arbitrary values instead of the declared tokens. Replace with:
- `text-[1.75rem]` → `text-xl` (maps to `--font-size-xl: 1.75rem`)
- `text-[2.25rem]` → `text-4xl` (maps to `--font-size-4xl: 2.25rem`)

The `text-2xl` in `ResultatCard.tsx` (quote decorators at lines 51 and 53) should move to `text-xl` or use a CSS approach. `text-2xl` is not in the 4-role type scale (`text-sm`, `text-base`, `text-xl`, `text-4xl`). At this decorative role, `text-xl` is sufficient.

### Spacing Token Correction

The `left-[-2rem]` arbitrary value in `mon-histoire/page.tsx:68` (timeline dot) should be replaced with `-left-8`. In Tailwind v4, spacing utilities follow the same pattern: `-left-8` = -32px, which matches the existing `-2rem` value exactly.

### Accessibility: Focus Rings on Cards

`ServiceCard`, `ProgrammeCard`, and `ResultatCard` are `<div>` elements with hover effects but no keyboard focus state. Since these are not interactive elements (no `onClick`, no `href`), they do not receive keyboard focus by default — they are not technically broken for keyboard navigation. However, adding `tabIndex={0}` + `focus-visible:ring-2 focus-visible:ring-bleu-electrique` would make them keyboard-discoverable per the spec.

**Decision for planner:** The spec states focus ring should use `bleu-electrique`. The audit flags this as a gap. The simplest fix is adding `focus-visible:ring-2 focus-visible:ring-bleu-electrique focus-visible:outline-none` to the card wrapper `<div>` — no `tabIndex` needed unless cards will ever be interactive.

### Title Template Separator Fix

`src/app/layout.tsx:27` uses `template: '%s | Antoine Profit'` (pipe) while the spec declares `'%s — Antoine Profit'` (em-dash). Fix:

```typescript
// src/app/layout.tsx line 27
template: '%s — Antoine Profit',
```

This affects all page `<title>` tags that use the template.

## Common Pitfalls

### Pitfall 1: error.tsx Must Be a Client Component
**What goes wrong:** Creating `error.tsx` without `'use client'` causes a build error.
**Why it happens:** Next.js requires error boundaries to be Client Components — they must use React state to catch errors thrown during rendering.
**How to avoid:** Always add `'use client'` as the first line of any `error.tsx` file.
**Warning signs:** Build error "error.tsx is not a Client Component".

### Pitfall 2: error.tsx Placement Scope
**What goes wrong:** Placing `error.tsx` in a page subfolder (`/resultats/error.tsx`) only catches errors on that page, not the layout.
**Why it happens:** `error.tsx` does not catch errors thrown by the layout at the same level — only errors in children.
**How to avoid:** Place `error.tsx` in `src/app/(frontend)/` to catch all frontend page errors. The `(frontend)/layout.tsx` itself fetches from Payload — if that throws, a higher-level error boundary is needed. For safety, also wrap the `layout.tsx` Payload call in try/catch.
**Warning signs:** Error on layout fetch shows raw Next.js error, not custom error.tsx.

### Pitfall 3: loading.tsx and Layout Fetches
**What goes wrong:** `loading.tsx` in `(frontend)/` shows while page fetches run but NOT while layout fetches run. If `(frontend)/layout.tsx`'s `payload.findGlobal()` is slow, the user sees nothing.
**Why it happens:** `loading.tsx` wraps only the `{children}` slot of the layout, not the layout itself.
**How to avoid:** For the layout-level WhatsApp fetch, wrap in try/catch with a fallback (empty string for `whatsappNumber`). This is already partially done but should be hardened.
**Warning signs:** Loading state appears to "skip" — page appears blank then jumps to content.

### Pitfall 4: Tailwind v4 Custom Token Names Are Case-Sensitive
**What goes wrong:** Using `bg-Blanc-pur` or `bg-BLANC-PUR` finds no match — the utility class doesn't apply.
**Why it happens:** Tailwind v4 `@theme` variable names map directly to utility classes. `--color-blanc-pur` → `bg-blanc-pur`.
**How to avoid:** Always check `globals.css` for the exact token name casing.
**Warning signs:** Color doesn't apply; no Tailwind compilation error.

### Pitfall 5: Title Template Change Affects All Pages
**What goes wrong:** Changing the template separator from `|` to `—` in `layout.tsx` changes the `<title>` output for every page, including admin-adjacent pages if they share the root layout.
**Why it happens:** The root `layout.tsx` at `src/app/layout.tsx` applies to all routes.
**How to avoid:** Verify that only the `(frontend)` route group pages use `title: 'string'` metadata — not Payload admin routes. The `(payload)` layout at `src/app/(payload)/layout.tsx` likely overrides the root layout, so this change is safe.
**Warning signs:** Payload admin page title inadvertently changes.
